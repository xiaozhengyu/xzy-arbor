# XZY Arbor 后续工作路线图

## 文档目的

本文记录 XZY Arbor 在完成轻量导图 MVP 之后，建议继续推进的工作方向、优先级和拆分方式。当前推荐路线是：先保障数据安全与交付可靠性，再提升高频编辑效率，然后逐步补齐内容输出、产品完整度和工程质量。

本文不直接替代具体功能规格。进入任一阶段实施前，仍应按项目既定 Spec-Driven Development 流程，在 `docs/spec/` 下为对应功能补齐需求、接口和实施方案。

## 当前基线

当前版本已经完成轻量桌面导图的基础闭环：

- Windows 桌面应用骨架已经建立，使用 Tauri 2、React 19 和 TypeScript。
- 已支持固定右向树布局、节点创建、编辑、删除、折叠、备注和拖拽整理。
- 已支持 JSON 文件的新建、打开、保存和另存为。
- 已有树操作和文档校验的基础单元测试。
- 已完成 release 版文件权限修复和指针驱动拖拽稳定性修复。

因此，下一阶段不宜马上堆叠大量新功能，而应优先补齐"放心保存、放心编辑、误操作可恢复、发布可验证"的能力。

## 总体优先级

| 优先级 | 主题 | 核心目标 | 建议节奏 |
| --- | --- | --- | --- |
| P0 | 数据安全与交付可靠性 | 降低丢数据、坏文件、误操作、发布不可用风险 | 立即推进 |
| P1 | 高频编辑效率 | 让日常绘制导图更快、更顺手 | P0 稳定后推进 |
| P2 | 内容输出与复用 | 支持把导图带出应用、复用结构和分享成果 | P1 核心项完成后推进 |
| P3 | 视觉与产品完整度 | 提升观感、引导和类 MindManager 的基础体验 | 与 P2 交错推进 |
| P4 | 工程质量 | 控制复杂度、提高测试覆盖、规范发布流程 | 穿插进行 |

## P0：数据安全与交付可靠性

### 目标

让用户可以放心把真实导图文件交给应用管理。即使误操作、关闭应用、打开错误文件或保存失败，也应尽量避免不可恢复的数据损失。

### 建议任务

1. **撤销 / 重做**
   - 支持 `Ctrl+Z` 撤销最近的结构或内容改动。
   - 支持 `Ctrl+Y` 或 `Ctrl+Shift+Z` 重做。
   - 覆盖节点新增、删除、移动、重命名、备注、折叠和视图变更中的关键操作。
   - 初版可以限制历史步数，例如 50 步，避免内存无限增长。

2. **保存确认流程优化**
   - 当前新建、打开、关闭前已有未保存提示，但交互还偏工程化。
   - 将输入 `save`、`discard`、`cancel` 的提示改为明确按钮式决策。
   - 新建、打开、关闭三个流程使用一致的确认逻辑。
   - 保存失败时不允许继续执行会丢失当前内容的操作。

3. **文件错误与兼容提示增强**
   - 对无效 JSON、不支持版本、结构非法、读取失败和写入失败给出更具体的中文提示。
   - 在错误提示中说明当前导图是否保持不变。
   - 为未来文件格式升级预留清晰的版本提示文案。

4. **最近文件列表**
   - 记录最近打开或保存的导图文件路径。
   - 支持从入口快速打开最近文件。
   - 文件不存在或读取失败时从列表中移除或标记不可用。

5. **发布验证清单**
   - 固化每次打包前需要执行的检查：`npm test`、`npm run build`、Tauri build、打开保存手动验证。
   - 记录安装包输出路径、版本号确认、基础冒烟用例。
   - 后续可升级为 CI 或发布脚本。
   - 详细的版本管理策略和 CI 规划见 P4 任务 4（发布与版本管理）。

6. **窗口标题与文件名显示**
   - 窗口标题栏显示当前打开的文件名。
   - 未保存时在标题栏显示脏状态标记（如文件名前加 `*` 或 `●`）。
   - 让用户一眼知道自己在操作哪个文件、是否需要保存。

### 验收口径

- 用户误删节点后可以通过撤销恢复。
- 保存失败不会导致当前内存文档被清空或替换。
- 打开坏文件时，当前文档保持不变，并看到明确错误提示。
- 可以从最近文件列表快速打开之前编辑过的导图文件。
- 每次 release 前有明确、可重复执行的验证清单。
- 窗口标题实时反映文件名和未保存状态。

### 风险与取舍

- 撤销 / 重做需要明确哪些状态进入历史栈。建议第一版聚焦文档内容与结构，视图平移缩放可以暂不进入历史。
- 最近文件涉及本地持久化，需选择 Tauri store、配置文件或简单本地文件方案，实施前应单独写规格。

## P1：高频编辑效率

### 目标

让用户在键盘和鼠标之间少切换，能快速创建、定位、整理和查找节点，使应用真正适合日常高频使用。

### 建议任务

1. **方向键导航**
   - 支持上下选择相邻可见节点。
   - 支持左右在父子层级之间移动选择。
   - 折叠节点时，右键可展开；展开节点时，左键可折叠或回到父节点。

2. **快捷键体系补齐**
   - `Ctrl+S` 已支持保存，后续补齐新建、打开、另存为、搜索、备注开关等快捷键。
   - 增加快捷键帮助入口，降低记忆成本。
   - 避免快捷键和文本编辑、备注输入冲突。

3. **复制、剪切、粘贴节点**
   - 支持复制当前节点及其子树。
   - 支持剪切并移动到目标节点下。
   - 粘贴时为新节点生成新 ID，避免文件结构冲突。
   - 初版可先支持应用内复制，不必打通系统剪贴板格式。

4. **快速搜索与定位**
   - 支持按节点文本搜索。
   - 搜索结果可以逐个跳转并自动选中节点。
   - 如果匹配节点位于折叠子树中，应提示或自动展开路径。

5. **节点聚焦与画布定位**
   - 提供"定位到选中节点"操作。
   - 创建、搜索跳转、打开文件后可将关键节点带入视野。
   - 解决大导图中选中状态和视口脱节的问题。

6. **批量展开 / 折叠**
   - 支持展开全部、折叠全部、折叠到某一层级。
   - 对大型导图可明显提升整理效率。

### 验收口径

- 用户可以主要通过键盘完成新增、选择、编辑、保存和查找。
- 大导图中能快速定位节点，不需要手动拖动画布寻找。
- 复制粘贴不会产生重复 ID 或非法树结构。
- 搜索折叠节点时不会出现"明明存在但找不到"的体验。

### 风险与取舍

- 方向键导航依赖当前可见布局顺序，应优先复用布局结果，避免单独维护一套不一致的遍历逻辑。
- 复制粘贴如果接入系统剪贴板，会引入跨应用格式设计。建议先做应用内复制。
- 搜索结果高亮、定位和自动展开可能拆成多轮实现，先保证可用，再优化体验。

## P2：内容输出与复用

### 目标

让用户能把导图成果带到其他场景中使用，例如文档、汇报、归档和二次编辑。

### 建议任务

1. **导出 PNG**
   - 输出当前导图可见区域或完整导图图片。
   - 支持透明或白底背景的取舍需在规格中明确。
   - 优先满足"插入文档 / 汇报"的场景。

2. **导出 SVG**
   - 保留矢量清晰度，适合文档和后续编辑。
   - 需要确保节点样式、文字和连线能正确转换。

3. **导出 Markdown 大纲**
   - 将树结构导出为 Markdown 层级列表。
   - 节点备注可作为子段落或可选附加内容。
   - 适合从导图转成计划、文章提纲或会议纪要。

4. **导入 Markdown 大纲**
   - 从缩进列表或标题层级生成导图。
   - 初版应限制格式，避免支持过多 Markdown 语法。

5. **模板与示例文件**
   - 提供常用结构模板，例如项目计划、读书笔记、会议记录、问题拆解。
   - 提供示例文件帮助用户理解应用能力。

6. **打印 / 截图友好模式**
   - 隐藏工具栏、状态栏和备注面板。
   - 使用适合截图的背景和边距。

### 验收口径

- 用户可以把导图导出为至少一种图片格式并插入其他文档。
- Markdown 导出结果层级正确，能被常见编辑器读取。
- 模板能帮助用户快速开始，而不是每次从空白导图开始。

### 风险与取舍

- PNG 和 SVG 导出涉及渲染一致性，尤其是 HTML 节点和 SVG 连线混合渲染。实施前需要明确导出技术路线。
- Markdown 导入必须限制输入格式，否则会变成复杂解析器项目。
- 模板不宜过多，第一版 3 到 5 个高频模板即可。

## P3：视觉与产品完整度

### 目标

在稳定和效率基础上，提升应用的可读性、引导性和长期使用舒适度。

### 建议任务

1. **主题与配色**
   - 支持浅色默认主题的细化。
   - 后续可考虑深色主题。
   - 节点、连线、选中态和拖拽态需要统一视觉语言。

2. **节点样式能力**
   - 支持基础颜色或强调样式。
   - 初版可以只支持少量预设，不引入任意 CSS 或复杂样式编辑。

3. **大纲视图**
   - 以树形列表展示全部节点。
   - 支持点击大纲项定位到画布节点。
   - 可作为搜索、批量整理和导出 Markdown 的辅助入口。

4. **备注体验增强**
   - 备注面板支持更清晰的当前节点上下文。
   - 可以增加备注存在状态、字数提示或清空确认。
   - 继续保持纯文本，暂不做富文本。

5. **空状态与新手引导**
   - 新建导图后给出最少量的操作提示。
   - 首次启动说明快捷键和保存方式。
   - 避免用长篇说明打断用户。

### 验收口径

- 新用户能在不阅读外部文档的情况下完成基础导图编辑和保存。
- 大导图阅读时节点层次更清晰，选中、拖拽和折叠状态更容易辨认。
- 可以通过大纲视图浏览全部节点，并点击大纲项快速定位到画布对应节点。
- 备注和状态栏能提供必要上下文，但不干扰画布空间。

### 风险与取舍

- 视觉能力容易膨胀为完整主题系统。建议先做预设样式，不做自由样式编辑。
- 大纲视图会占用空间，需要和备注面板、未来搜索面板统一规划侧栏。
- 新手引导应轻量，不要变成必须点击完成的教程流程。

## P4：工程质量与长期维护

### 目标

在功能增长前控制复杂度，确保后续迭代不会被单文件状态、测试不足或发布流程不清晰拖慢。

### 建议任务

1. **拆分 App 状态与交互职责**
   - 当前主应用组件承担状态、快捷键、文件操作、拖拽、画布和渲染等多种职责。
   - 建议逐步拆分为画布组件、节点组件、备注面板、工具栏和状态管理 hooks。
   - 拆分应服务于具体功能，不做一次性大重构。

2. **补齐纯函数测试**
   - 增加布局计算测试，覆盖折叠、层级、边连线和隐藏数量。
   - 增加更多文档校验测试，覆盖非法 viewport、非法节点字段、非法 children。
   - 增加移动边界测试，覆盖根节点、同级前后移动和不存在节点。

3. **文件边界测试策略**
   - 将 Tauri 文件 API 包装层保持薄而清晰。
   - 对解析、序列化、错误码映射做可测试拆分。
   - 不强行 mock 桌面系统能力，优先测试纯逻辑。

4. **发布与版本管理**
   - 明确应用版本号和安装包版本号同步方式。
   - 建立 release checklist。
   - 后续可考虑 GitHub Actions 或本地发布脚本。

5. **文档维护机制**
   - 行为变更前先补 `docs/spec/` 三件套。
   - 实现后更新 `docs/decisions/AI_CHANGELOG.md`。
   - 重要设计取舍记录在对应 spec 或 decision 文档中。

6. **依赖与安全检查**
   - 定期执行 `npm audit`。
   - Tauri 权限保持最小化，只开放文件操作需要的能力。
   - 新增依赖前评估必要性、包体影响和维护状态。

7. **大数据量性能监控**
   - 在功能稳定后对 100+、500+ 节点规模做性能基准测试。
   - 确认布局计算、渲染和交互在高节点数下的瓶颈。
   - 根据结果评估是否需要虚拟化渲染或增量布局优化。

### 验收口径

- 新功能不需要继续扩大单个主组件的复杂度。
- 树、布局、文档校验等核心逻辑有稳定测试覆盖。
- 每次发布前的步骤清晰，可重复执行。
- 文档能跟随行为变化同步更新。
- 500 节点以下导图交互无明显卡顿。

### 风险与取舍

- 工程质量不应脱离功能目标独立大改。建议每个阶段顺手拆解正在触碰的代码区域。
- UI 自动化测试成本较高，初期可先加强纯函数和边界逻辑测试。
- CI 和发布脚本可以在发布频率变高后再投入，不必第一时间完成。
- 性能优化应在出现可感知问题后再深入，不必过早优化。

## 推荐执行顺序

### 第一轮：P0-A 数据安全最小闭环

1. 撤销 / 重做。
2. 保存确认流程改为按钮式交互。
3. 打开坏文件时保证当前文档不变并提示清楚。
4. 窗口标题显示文件名和未保存状态。
5. 补充针对撤销/重做、保存确认、错误处理等新增逻辑的树操作、文档校验和交互边界测试。

这一轮完成后，用户即使误删、误拖或打开错误文件，也更容易恢复。

### 第二轮：P0-B 便利性收尾

1. 实现最近文件列表，包括本地持久化和入口 UI。
2. 固化和验证发布检查清单，确保每次 release 可重复执行。

这一轮完成后，P0 数据安全与交付可靠性全部闭环。

### 第三轮：P1-A 键盘效率

1. 方向键导航。
2. 快捷键帮助入口。
3. 选中节点定位到视口。
4. 展开 / 折叠快捷操作。

这一轮完成后，日常编辑效率会明显提高。

### 第四轮：P1-B 查找与复制

1. 搜索节点文本。
2. 搜索结果跳转和自动展开路径。
3. 应用内复制、剪切、粘贴节点。
4. 粘贴时生成新 ID 并保留子树结构。

这一轮完成后，大导图整理能力会明显提高。

### 第五轮：P2 输出能力

1. Markdown 导出。
2. PNG 或 SVG 导出二选一先做。
3. 示例文件和 3 到 5 个模板。
4. 截图友好模式。

这一轮完成后，导图可以更方便地进入文档、汇报和归档流程。

### 第六轮：P3 与 P4 持续打磨

1. 主题和节点样式预设。
2. 大纲视图。
3. 备注体验增强。
4. 主组件职责拆分。
5. 版本号管理、CI 脚本和发布流程固化。
6. 大数据量性能测试与优化。

这一轮适合在核心稳定性和效率体验已经可用后推进。

其余 P4 任务（补齐纯函数测试、文件边界测试、文档维护、依赖与安全检查）属于穿插性工作，应在各轮次开发过程中顺手推进，不集中到单一轮次。例如：每轮新增功能时同步补齐对应测试，每次行为变更时同步更新 `docs/spec/` 和 `AI_CHANGELOG.md`，定期执行 `npm audit`。

## 后期待选功能

以下功能目前不纳入近期计划，但保留为后续可选方向，待核心体验稳定后根据实际需求评估：

- **多标签 / 多文件管理**：同时打开多个导图文件，在标签页之间切换。涉及架构层面调整（文档状态从单例变成列表），评估时需考虑对现有单文档模型的改造范围。
- **节点图标 / 标记**：优先级标记、状态图标、旗标、星标等预设图标，提升导图信息密度和可读性。
- **关系连线**：非父子节点之间的关联线，标注跨分支关系（如"A 影响 B"）。实现涉及布局层和渲染层改动。
- **导入其他导图格式**：支持导入 XMind、FreeMind、MindManager 等常见导图格式，降低用户迁移成本。
- **任务管理能力**：节点上的复选框、进度条、截止日期等，让导图兼做轻量任务拆解工具。会改变数据模型，需慎重评估。

## 暂不建议立即做的事项

- 云同步、账号体系和多人协作：会显著改变架构，不符合当前轻量桌面工具定位。
- 富文本备注：会增加存储格式、编辑器和导出复杂度，当前纯文本更稳。
- 自由画布定位：与固定右向树的核心设计相冲突，会显著增加布局和拖拽复杂度。
- 插件系统：当前还没有足够稳定的扩展边界，过早设计会增加负担。
- 大而全的主题编辑器：建议先做少量预设，避免变成样式配置工具。
- 命令面板：用户偏好鼠标操作，不需要快捷键统一入口。
- 桌面端自动更新：当前手动发布频率低，自动更新机制投入产出比不高。
- 无障碍基础：当前仅服务个人用户，暂无 a11y 合规需求。
- 演示模式：当前使用场景不涉及汇报展示。
